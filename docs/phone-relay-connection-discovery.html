Product architecture memo

Connection should be adb-first. Discovery should only help.

Phone Relay should feel automatic, but the reliable source of truth is still the live adb transport list. USB, Wi-Fi, mDNS, and saved routes each have a role, but only proven adb readiness should start or recover a mirror.

The core principle

The app can be helpful without being speculative. Phone Relay should make the best path obvious, do background work quietly, and avoid changing the active mirror unless a transport has been proven ready.

Source of truth

adb decides what is real

A device is mirror-ready only when adb sees it and it responds to a lightweight command.

Discovery role

mDNS finds possibilities

Bonjour can suggest phones for pairing or recovery, but it should not launch mirroring by itself.

Product layer

Saved routes need proof

A remembered Wi-Fi address is useful only when it passes connect and shell-readiness checks.

Workflow

The normal connection path should be short and deterministic. Deeper discovery and recovery should exist, but they should be invoked only when the user asks or when USB disappears and a prepared Wi-Fi path is already proven.

Poll live adb devices

Start with `adb devices -l`. Authorized USB and already-connected Wi-Fi transports are the first-class candidates.

Prefer USB for immediate start

If an authorized USB phone is present, start mirroring quickly and do not make the user wait for Wi-Fi preparation.

Prepare Wi-Fi in the background

When enabled, run USB-to-Wi-Fi handoff as a speculative job. It should never block or destabilize the current USB mirror.

Verify before saving

Persist a Wi-Fi route only after adb connect succeeds and the device answers a shell probe over that Wi-Fi serial.

Use prepared Wi-Fi on reconnect

When the cable is removed or the mirror needs recovery, switch to the prepared route if it is still live or can be verified quickly.

Escalate only on explicit action

Manual reconnect can try saved routes and mDNS rediscovery. The background path should avoid silent stale-route dialing.

System roles

Each signal has a clear job. The UX gets simpler when the app does not treat every signal as equally authoritative.

Layer 1

Live adb

Primary source for connection ownership. If adb sees a transport and shell works, the phone is usable.

Layer 2

Prepared Wi-Fi

A verified standby route. It improves continuity without replacing the active USB mirror mid-session.

Layer 3

Discovery and memory

mDNS and saved routes help the app find candidates, but every candidate must still become live adb.

Benefits

This model keeps the app predictable for engineers and understandable for users while still enabling a seamless “plug in once, continue over Wi-Fi later” experience.

User experience

Fast first mirror

USB starts immediately instead of waiting on Wi-Fi discovery, local network permissions, or stale saved state.

Reliability

Fewer invisible loops

Background auto-connect does not keep dialing dead addresses and pinning the UI in a vague connecting state.

Continuity

Wi-Fi feels prepared

If handoff succeeds, unplugging can feel natural because a verified Wi-Fi route is already waiting.

Supportability

Status is explainable

Diagnostics can say USB active, Wi-Fi preparing, Wi-Fi ready, or Wi-Fi failed instead of hiding the real state.

Non-negotiable guarantees

These invariants keep the model from regressing. Each came from a real failure on real hardware, so they are requirements, not preferences — a rewrite that drops one reintroduces a bug we already fixed.

1
“Ready” means data is flowing, not that connect returned OK. adb connect succeeds against half-open and stale endpoints, so readiness requires a shell probe that actually answers. Once mirroring, TCP keepalive plus a ~5s data-stall watchdog must keep watching, so a silent Wi-Fi drop ends the session in seconds instead of freezing on the last frame.
2
Separate “blocked by macOS” from “phone offline.” Repeated “No route to host” is usually a macOS Local Network permission denial — and it silently resets on every ad-hoc re-sign — not an absent phone. Surface it as a distinct action (grant Local Network access) in diagnostics, quietly, while USB keeps working. Never collapse it into a generic offline error.
3
Bound the retries; break crash loops. Some device servers load-then-bail (e.g. a Samsung aborting when the client requests display-power-off). Auto-reconnect must latch one attempt per device state, back off after quick failures, and stop — never hammer a crashing server or alternate between transports (route-key flapping is itself a storm).
4
A USB serial must never overwrite a saved Wi-Fi IP. Reconnecting over the cable must preserve the remembered host:port, because that is what launch and presence reconnect dial once the cable is gone. A bare USB session has no wireless address of its own to contribute.
5
Subnet match is a hint; reachability is the gate. Comparing the Mac and phone subnets only chooses where to prime the route. AP/client isolation means same-subnet can be unreachable and different-subnet can still work, so only an actual connect-plus-shell probe is allowed to authorize a handoff.
6
Serialize mutating adb commands. connect/pair/tcpip run under one lock so concurrent USB verification and Wi-Fi probing can’t race the adb server; read-only queries stay unlocked so the UI stays responsive.

Rules and risks

The risky part is not Wi-Fi itself. The risky part is letting stale or partial signals take control of the mirror.

✓
Do not switch mid-session just because Wi-Fi appears. Keep a stable USB mirror unless the user unplugs or explicitly asks to reconnect.
✓
Do not trust mDNS as readiness. mDNS means a service is visible. It does not prove adb is authorized, awake, or usable.
✓
Do not save unverified Wi-Fi routes. Only persist addresses that answered over adb, otherwise future reconnects inherit bad state.
✓
Do not interrupt healthy USB with Wi-Fi errors. Failed handoff should be quiet while USB works, with detail available in diagnostics.
✓
Do back off after repeated Wi-Fi failures. Retries should be bounded so background work does not become battery, CPU, or adb churn.
✓
Do respect explicit user intent. Manual disconnect, stop mirroring, pairing, and device switching should cancel hidden reconnect work.
